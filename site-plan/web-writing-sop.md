# 網頁製作 SOP

Last updated: 2026-06-02

## 目的

這份 SOP 記錄本專案撰寫與更新教學網站頁面時要遵守的固定規則。

## 提示詞複製區

凡是在網站頁面中提供「提示詞」「可貼給 AI 的文字」「給 AI 助手看的內容」讓使用者複製時，一律要提供一鍵複製按鈕。

## 對同仁說話的口吻

網站頁面是給校內行政伙伴、行政同仁看的，語氣要像同事之間的引導，不要像工程文件、產品說明或維護者交接。

核心說法優先使用：

```text
AI 先整理，自己要確認。
```

使用原則：

- 一般頁面避免使用翻譯腔的「human in the loop」「人再確認」。
- 面向同仁的文案優先寫「自己要確認」「等我確認」「確認後再使用」。
- 技術或高風險流程仍可提到 dry-run、備份、確認清單，但要說清楚同仁要看什麼、確認什麼。
- 不要把確認責任寫得像抽象制度；要讓同仁知道正式公告、寄信、送簽、寫入系統前，最後要自己確認。

### 網站既有寫作風格

目前網站的主線風格是：同事式、低技術門檻、先做一件事。

寫頁面時要讓同仁感覺「我知道下一步要做什麼」，而不是「我需要先懂工具」。主頁面和學習路線優先使用白話、短句、動作導向標題。

常見句型：

- 「先……，再……」
- 「不用先懂……」
- 「複製這段貼給自己的 AI 助手」
- 「AI 可以協助我們，但自己一定要確認」
- 「正式送出、公告、寄信、寫入系統前，先停下來」

標題寫法：

- 優先寫使用者要做的事，例如「先安裝 Codex」「文件整理成文字檔」「讓 AI 使用瀏覽器前，先確認五件事」。
- 避免在面向同仁的標題中先放工具名、repo 名、協定名或工程縮寫。
- 若工具名稱不可避免，先用白話說明它能幫同仁完成什麼。

段落寫法：

- 一段只說一個重點。
- 可以一句話講清楚，就不要重複講。
- 不要用不同標題、卡片、提醒框反覆講同一個原則；如果內容只是換句話說，應合併或刪除。
- 先說為什麼要做，再說怎麼開始。
- 遇到技術設定，改寫成同仁能確認的問題：哪個帳號、哪個網站、哪些資料、哪些動作。
- 不要把維護者的討論脈絡放進一般頁面，例如底層架構、GitHub repo、指令、port、session、profile、cookie、API 細節。

一般使用者入口規則：

- 首頁、學習路線、單元主頁的主要按鈕應優先帶到網站單元頁。
- 不要在一般使用者主線入口直接放 GitHub repo、流程包材料、指令速查或維護者文件。
- 技術材料可放在維護者資源頁、flow-pack、repo README，或單元頁最末端的進階備查；但一般同仁必須先能只靠單元頁完成理解。
- 如果頁面需要請 AI 參考外部 repo 或流程包，提示詞要說明「AI 要拿它做什麼」，不要只丟連結給使用者。

### 角色定位與自足性

網站上的可複製提示詞，是給「承辦人的 AI 助手」看的，不是給本專案維護者看的，也不是給已經參與本網站規劃討論的 AI 看。撰寫時必須假設接收提示詞的 AI 助手不知道本專案背景、不知道前一次對話、不知道我們討論過哪些設計決策。

提示詞必須自足：

- 開頭先交代 AI 的角色，例如「你是協助學校行政承辦人的 AI 助手」。
- 說清楚使用者身份、任務目標、限制條件、輸出格式、何時要停下來等確認。
- 需要引用外部 repo 或流程包時，要說明 AI 要拿它做什麼，不要只丟連結。
- 不要把維護者討論脈絡寫進提示詞，例如「我們剛剛討論」「依照本網站第幾單元」「如前面所說」「這個單元要呈現」。
- 不要假設 AI 已讀過網站其他頁面。若任務需要前置處理，提示詞要用一般語言說明，例如「如果資料是 PDF、Word、圖片或掃描檔，請先協助轉成可檢查的文字」，不要寫成「請接第 2 單元」。
- 網站頁面可以在提示詞之外提供教學脈絡；但可複製提示詞本身要能被承辦人單獨貼到另一個 AI 對話中使用。

撰寫後要檢查可複製提示詞內是否出現下列不當脈絡字，除非是任務資料本身的一部分：

- 「我們討論」
- 「本網站」
- 「這個單元」
- 「第 X 單元」
- 「如前所述」
- 「流程包材料」但沒有說明用途

實作規則：

- 使用 `.prompt-box` 包住提示詞區塊。
- 使用 `.prompt-header` 放標題與複製按鈕。
- 複製按鈕文字使用「複製」。
- 複製按鈕加上 `data-copy-prompt`。
- 提示詞本文放在 `<pre><code>...</code></pre>`。
- 頁面需載入 `../copy-prompts.js`。

範例：

```html
<div class="prompt-box">
  <div class="prompt-header">
    <p class="prompt-title">可直接貼給 AI 的句子</p>
    <button class="copy-button" type="button" data-copy-prompt>複製</button>
  </div>
  <pre><code>請把這段文字貼給 AI。</code></pre>
</div>
<script src="../copy-prompts.js"></script>
```

檢查規則：

- 每個 `.prompt-box` 都要有 `button[data-copy-prompt]`。
- 每個含 `button[data-copy-prompt]` 的頁面都要載入 `copy-prompts.js`。
- 更新後要跑站內連結檢查。

安裝或環境設定類提示詞：

- 給人看的說明要提醒使用者：若 AI 有本機操作或安裝權限，可以請 AI 代為檢查與安裝。
- 給 AI 的提示詞要明確要求：有權限就代辦；沒有權限就用使用者看得懂的步驟引導，不要只提供技術文件連結。
- 若規則已經寫在 repo、skill 或 flow-pack 裡，網站提示詞只負責指向正確 repo、授權 AI 檢查安裝，並要求 AI 以 repo 規則為準；不要在提示詞裡重抄整套技術規則。
- 若多個單元都只是「安裝 repo 後由 AI 自行判斷使用」，優先在首頁提供一段整合安裝提示詞，一次列出所有基礎工具 repo；各單元頁可保留單一工具提示詞作為備用入口。
- 若提示詞要求 AI 使用某個工具、安裝某個工具、或檢查環境中是否缺少工具，一律加入下列標準句：

```text
如果你有本機操作或安裝權限，請代我檢查並完成必要安裝；如果你沒有權限，請用我看得懂的步驟帶我完成，不要只丟技術文件連結給我。
```

## Markdown 材料連結

網站頁面不要直接連到本機 repo 內的 `.md` 檔案。

原因：

- `.md` 不是網站頁面，一般使用者點開後會看到原始 Markdown。
- 本機 HTTP server 可能沒有正確提供 UTF-8 charset，瀏覽器會把中文顯示成亂碼。

實作規則：

- 給一般使用者看的內容，優先做成 `site/modules/*.html` 網頁。
- 若需要連到完整流程包資料夾，連到 GitHub 的 `tree/main/...`。
- 若需要連到單一 Markdown 模板或檢查清單，連到 GitHub 的 `blob/main/...`。
- 網站 HTML 中不應出現指向本機相對路徑的 `.md` 連結，例如 `../flow-packs/.../README.md`。

檢查規則：

- 更新後搜尋 `site/` 中是否還有 `href` 指向本機 `.md`。
- 允許外部 GitHub `blob/main/*.md` 或 `tree/main/...` 連結，因為 GitHub 會用網頁方式呈現。

## 工具型流程頁寫法

如果某個單元本質上需要工具、指令、套件或環境檢查，網站頁面仍然要先從一般使用者的最小用途寫起，不要先列工具名。

寫作順序：

1. 先說使用者要完成什麼事。
2. 再說使用者要準備什麼資料。
3. 把工具選型、安裝、指令與錯誤處理放進給 AI 的提示詞。
4. 若流程有失真風險，例如文件轉換、資料清理或批次操作，要在頁面上用白話提醒「完成後要檢查什麼」。
5. 完整流程、指令速查與檢查清單放在進階備查，連到 GitHub 渲染頁。

文件轉換類頁面特別要提醒：

- 不要把真實公文、學生或家長資料、未公開資料直接上傳到不明外部服務。
- 轉換成功不等於內容正確，還要檢查標題、表格、中文、漏頁與頁首頁尾。
- 掃描檔或沒有文字層的 PDF 需要 OCR 或人工確認，不能假裝已完整轉換。

## 成果型工作流頁寫法

如果某個單元的目的不是安裝工具，而是產出一份看得見的成果，例如簡報、公告、報告或圖像，網站頁面不要把它寫成工具安裝頁。

寫作順序：

1. 先說使用者最後要完成什麼成果。
2. 說明使用者最少要準備哪些資料。
3. 提供一段可複製提示詞，請 AI 依照 flow-pack 或 repo 內材料帶使用者完成。
4. 若規則已經寫在 flow-pack 裡，提示詞只需要指向 GitHub 材料並要求以 flow-pack 規則為準。
5. 頁面要清楚分出 AI 可協助的部分與人必須確認的部分。
6. flow-pack 材料作為進階備查，連到 GitHub `tree/main/...` 或 `blob/main/...`。
7. 這類單元不要加進首頁整合安裝提示詞；首頁只放需要安裝或套用的基礎工具 repo。

行政簡報類頁面特別要提醒：

- AI 可以協助整理 brief、大綱、圖像需求與審查清單。
- 人要確認事實、數字、姓名、日期、照片授權、個資與對外分享界線。
- 正式校徽、頁碼與固定識別使用正式素材或 overlay，不讓影像模型自由生成。
- 成果完成後，請 AI 回寫成下次可重複使用的小 SOP。
