# 第二單元發展紀錄：文件轉 Markdown

Last updated: 2026-06-03

## 單元定位

「文件轉 Markdown」是第一個支援型成果流程。它不是要一般使用者學會某個轉檔工具，而是讓使用者在把文件交給 AI 摘要、整理或改寫之前，先得到一份乾淨、可檢查、可再利用的文字版本。

對一般使用者來說，這個單元的核心用途只有一件事：

> 複製提示詞給 AI，請 AI 幫忙把文件變成乾淨文字，並檢查結果能不能安心使用。

主句採用：

> 這是把文件變成 AI 看得懂的乾淨文字用的流程。

## 最小用途判斷

一般使用者不需要先理解 MarkItDown、doc2md-toolkit、pandoc、OCR、文字層或命令列參數。

使用者需要知道的是：

- 我有一份文件或一個資料夾。
- 我想讓 AI 摘要、整理成會議紀錄、做簡報 brief 或改寫公告。
- 直接把原始檔交給 AI 可能不穩，因為表格、頁碼、掃描檔、亂碼或漏頁會讓結果失真。
- 所以先請 AI 轉成乾淨文字，再用清單檢查。

因此網站頁面把主操作改成「複製提示詞」，而不是「下載流程包」或「閱讀指令速查」。

## 人與 AI 的分工

網站模組頁給人看：

- 用白話說明為什麼要先轉成乾淨文字。
- 告訴使用者要準備文件路徑或資料夾路徑。
- 提供可直接貼給 AI 的提示詞與複製鈕。
- 用三張卡提醒轉之前、轉之後、再利用。

提示詞給 AI 看：

- 判斷文件類型。
- 檢查敏感資料風險。
- 優先檢查這台電腦是否已經可以使用 doc2md-toolkit 或 `doc2md` 指令；若尚未安裝，再從 GitHub repo 取得或引導使用者安裝。
- 一般文件走 MarkItDown，中文 PDF、直行教材或一字一行抽文字問題走 pdf2txt。
- 不預設安裝或使用重型 OCR 工具。
- 10 頁以內的掃描 PDF、圖片型文件或沒有文字層的 PDF，可轉成逐頁圖片後，用小量 OCR 或 AI 視覺讀取救援。
- 超過 10 頁或版面複雜時，不硬做，標記為需要正式 OCR 或人工確認。
- 大量掃描或複雜文件若沒有個資、機密或未公開資料，可把 NotebookLM 作為雲端閱讀理解式替代路線。這不是本機 OCR，也不是預設流程；使用者必須先確認資料可上傳。
- NotebookLM 不放在主要可複製提示詞內，避免一般使用者把雲端上傳誤認為預設流程；只放在頁面下方備註與安全說明。
- 未來可開發 NotebookLM CDP-assisted route：使用者先登入、上傳來源並確認資料邊界後，腳本只協助貼提示詞、等待回覆、擷取 Markdown 與產生待確認清單。
- 執行轉換。
- 判斷掃描 PDF 或沒有文字層時是否需要 OCR 或人工確認。
- 檢查標題、表格、中文、漏頁、頁首頁尾與附件。
- 輸出轉換結果摘要與待確認事項。

flow-pack materials 給 AI 或維護者備查：

- 完整流程。
- 指令速查。
- 品質檢查清單。
- 安全注意事項。

## 本次網站調整

`site/modules/doc2md.html` 已重寫為第二單元的使用者入口：

- 頁首主句改成「這是把文件變成 AI 看得懂的乾淨文字用的流程。」
- 主要 CTA 改為給 AI 的安裝/套用提示詞。
- 提示詞加入工具缺失標準句：

```text
如果你有本機操作或安裝權限，請代我檢查並完成必要安裝；如果你沒有權限，請用我看得懂的步驟帶我完成，不要只丟技術文件連結給我。
```

- 頁面載入 `../copy-prompts.js`，提示詞區塊有「複製」鈕。
- 進階材料連結一律連 GitHub 渲染頁或 GitHub 資料夾，不連本機 `.md`。
- 後續再精簡主提示詞：不重抄 doc2md-toolkit 的完整引擎分流、OCR 分流與檢查清單，也不要求 AI 在安裝提示詞裡立刻轉檔；只要求 AI 安裝或套用 repo，並依 README / SKILL.md / workflow / checklist 執行。真正轉檔留到使用者下一步提供文件時，由 skill 自行判斷。
- 後續再把第一、二單元整合到首頁的一段「基礎工具安裝提示詞」：一次安裝或套用 `windows-powershell-encoding-skill` 與 `doc2md-toolkit`。未來若新增同類型 repo 工具，優先加到首頁整合提示詞。
- 後續決定不把 MinerU 或其他重型 OCR 工具放入主流程；第二單元預設只使用 doc2md-toolkit 中的 MarkItDown 與 pdf2txt。
- 後續補上小量例外：10 頁以內的掃描或圖片型文件，可以由 AI 轉成逐頁圖片後做小量 OCR/視覺讀取；這是救援流程，不是預設主流程。
- 後續補上 NotebookLM 進階路線：大量或複雜但可上傳雲端的文件，可以由使用者自行上傳 NotebookLM，借用其閱讀理解能力整理成 Markdown、摘要或章節大綱。若含個資、機密、未公開公文或校內敏感資料，不建議上傳。這段應作為下方備註，不放進主要轉換提示詞。
- NotebookLM CDP-assisted route 只記為未來可開發方向。腳本不得自動登入、不得自行選取敏感文件、不得保存 cookies/profile/session；只能在使用者已登入、已上傳來源、已明確同意後協助貼提示詞、等待回覆、擷取結果與輸出檢查清單。

NotebookLM 參考邊界：

- NotebookLM 來源是使用者匯入或上傳的靜態副本。
- NotebookLM 可加入 PDF、Google Docs/Slides、網站、YouTube 等來源。
- 一般帳號有來源數、字數與上傳大小限制，限制可能隨服務調整。
- Notes 可匯出成 Google Docs 或 Sheets，但匯出檔不會與原 note 同步。
- 若使用個人 Google 帳號並提供回饋，內容可能涉及 Google 的疑難排解、濫用防治或改善服務流程；因此真實個資與未公開資料不應放入公開或不適合的雲端流程。

## 後續單元可複用原則

文件整理類、資料清理類、匯入匯出類流程都可沿用這個模式：

1. 人只需要知道要準備什麼資料與最小目的。
2. 工具選型與環境檢查放進 AI 提示詞。
3. 網頁主要操作是複製提示詞；提示詞負責指向 repo，不重抄 repo 內規則。
4. flow-pack 是備查，不是一般使用者的第一個入口。
5. 每個轉換流程都要有品質檢查，不只檢查是否產出檔案。
6. 每個涉及真實文件的流程都要先提醒敏感資料與公開分享限制。
